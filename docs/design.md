## Problem Statement

GPU availability information for AI training is dominated by pricing-comparison sites
(Spheron, GetDeploying, AIMultiple, Vantage, ec2instances.info) that treat provider
catalogs as ground truth. The result: SKUs that don't physically exist in-region appear
alongside ones that do. An H100 listed in GCP's `northamerica-northeast1` pricing page
is not the same as an H100 you can actually launch there. AWS `ca-central-1` lists P5
instances the default account cannot start. OVHcloud Beauharnois lists H100 catalog
entries while only V100 hardware physically ships.

Canada's AI sovereignty discussion suffers acutely from this gap. The policy discourse
(federal AI Compute Access Fund, Alliance Canada's sovereign compute vision, ISED's
AI Sovereign Compute Infrastructure Program) is qualitative and forward-looking. The
ground truth — which current-generation GPUs you can actually launch in a Canadian
region today — sits locked inside provider API responses no one publishes.

The dashboard closes that gap with auditable, continuously-updated, citation-ready data.

## What Makes This Cool

- **Phantom inventory exposure.** Every other comparison site trusts provider catalogs.
  This one reconciles "listed" against "actually launchable" via real probes.
- **Evidence-object architecture.** The primitive is the evidence record
  (timestamp, endpoint, request, response, verdict) written to an append-only log.
  Dashboard, leaderboard, reports are views on top.
- **Commit history as audit log.** Probes commit evidence JSON to the Git repo every
  30 minutes. Every data point has a cryptographic permalink via commit SHA. Providers
  cannot quietly retcon their catalogs; the history is public.
- **G7 framing.** "Canada ranks 6th of 7 G7 countries by per-capita current-gen GPU
  availability" is rhetorically sharper than OECD or EU framing for the Canadian policy
  audience.
- **Auto-generated citeable artifacts.** The dashboard is the live view; quarterly
  reports are auto-generated from the same evidence layer. Both are citation-ready.

## Constraints

- Open source. MIT license. Public repo from day one.
- Must stay inside free/hobby tiers — Vercel (hosting), GitHub Actions (cron + storage),
  and providers' unauthed + free-tier authed APIs.
- **No paid API keys** that require ongoing personal funding. This explicitly rules
  out any probe that creates billable cloud resources (e.g., DigitalOcean probe-create
  of a GPU droplet — the smallest GPU droplet is $2-4/hr). See §Probe Strategy below.
- No coordination with ISED or Alliance Canada. Independent of government.
- **V1 ship milestone = all four views functional at first public URL.** The views
  arrive incrementally during the implementation (see §Next Steps), but the launch
  announcement waits until all four are shipped. No "soft launch" of Phantom Wall alone.
- Methodology must be open source and reproducible — no black-box verdict logic.
- Canadian data residency not required (Vercel's edge network is fine; we're publishing
  observations, not handling regulated data).

## Current-Generation Training GPU Taxonomy (v1)

The Sovereignty Gap Index and G7 Leaderboard metrics both depend on this definition.
Inlined here so V1 implementation is unblocked; versioned for replay determinism.

**Included as "current-generation training" (as of 2026-04-20):**

| Vendor | Chip  | Variants                | Notes                                      |
|--------|-------|-------------------------|--------------------------------------------|
| NVIDIA | H100  | SXM 80GB, PCIe 80GB     | Widely deployed; baseline current-gen      |
| NVIDIA | H200  | SXM 141GB               | Memory-uplifted H100 successor             |
| NVIDIA | B100  | SXM                     | Blackwell training                         |
| NVIDIA | B200  | SXM                     | Blackwell training, full variant           |
| NVIDIA | GB200 | NVL72 rack-scale        | Counted per-GPU, not per-rack              |

| AMD    | MI325X | OAM                     | Training-capable Instinct successor        |

- TPU v4/v5 (non-NVIDIA, excluded from v1 metric; revisit in v2)

Hyperscalers and Canadian-facing providers with verifiable APIs relevant to the

they don't factor into the sovereignty argument the dashboard makes.

4. OVHcloud (BHS Canada + G7 peer regions)
5. DigitalOcean (TOR1 Toronto + G7 peer regions where GPU offered)
6. **Oracle Cloud Infrastructure (OCI)** — Toronto (YYZ) + Montreal (YUL) + G7 peer

**V1 providers — catalog-only tier + quarterly manual attestation (Canadian-rooted,

7. **Hut 8 Corp (Hut8 Compute / HPC AI)** — Canadian-headquartered AI-infra company
   with Alberta and Ontario data centers. Public marketing page lists H100/H200

   fetch of the public pricing/availability page + quarterly email attestation
   logged as an evidence record.
8. **Iris Energy (IREN) — AI Cloud** — British Columbia operations (Prince George,

**Explicitly deferred to V2:**

- Vast.ai, RunPod, Lambda Labs, TensorDock, CoreWeave — marketplace / boutique
- IBM Cloud (ca-tor), Vultr (Toronto), Paperspace — second-tier commercial, included
  after V1 coverage-matrix stabilizes.

  verifiable catalog as of April 2026. Documented in "known gaps" on the
  Methodology page; re-evaluated quarterly.

**Non-commercial compute (excluded from metric, disclosed on Methodology page):**

- Alliance Canada (formerly Compute Canada) — academic/research allocation, not
  rentable.
- Mila / Vector / CIFAR-adjacent compute partnerships — institutional.
- ISED's AI Compute Access Fund allocations — grant-gated, not a commercial SKU.

These genuinely exist as Canadian GPU capacity. They are deliberately not counted
in the Sovereignty Gap Index because the dashboard's question is "GPUs a Canadian
company or researcher can go rent today," not "GPUs physically on Canadian soil."
The exclusion is surfaced explicitly on the Methodology page so a policy reader
isn't surprised.

## Probe Strategy — Signal Sources and Limitations

This subsection replaces hand-wavy "live launchability probing" with concrete signals
and honest limitations, so the verdict engine is implementable without guessing.

**AWS**

- Signal A (free): `DescribeInstanceTypeOfferings` with `LocationType=availability-zone`
  → definitive "is SKU X offered in AZ Y by AWS" (not account-specific).
- Signal B (free, account-scoped): `GetServiceQuota` for the P/G families → the
  account's ceiling. Does NOT equal inventory; still informative.
- Signal C (free, general): `GetSpotPlacementScores` → 1-10 score for "can I launch
  this in that AZ right now?" Best launchability proxy.
- **Known limitation:** Dry-run `CreateCapacityReservation` on a free-tier account
  will always return `InsufficientInstanceCapacity` for P5 in Canadian regions
  because the account has quota 0. This is documented as a false-positive risk on
  the Methodology page. Spot Placement Score is the ground-truth signal.

**Azure**

- Signal A (free, unauthed): Retail Prices API → full SKU catalog per `armRegionName`.
- Signal B (free, authed): `Microsoft.Compute/skus` with `restrictions[]` array —
  explicit `NotAvailableForSubscription` / `Location` restriction codes. Cleanest
  launchability signal of any hyperscaler.

**GCP**

- Signal A (authed): `acceleratorTypes.list` per zone → catalog.
- Signal B (authed, lightweight): `reservations.list` for visibility.
- **Known limitation:** Real-capacity probing via `instances.insert --validate-only`
  costs nothing but counts against API quotas; the cadence must stay below 5 qps/zone.

**OVHcloud**

- Signal A (free, unauthed): `ca.api.ovh.com/1.0/order/catalog/public/cloud` → full
  plan catalog including H100 codes for CA subsidiary.
- Signal B (free, fragile): Playwright-rendered scrape of the public pricing page's
  per-region stock badge. OVH redesigns frequently; scraper brittleness is expected.
  Broken-scraper alerts route to the methodology page's "known degradations" list.

**DigitalOcean**

- Signal A (free, authed): `/v2/sizes` catalog + `gpu_info` field.
- **Known limitation:** DO offers no dry-run droplet create. Probe-by-create-then-destroy
  costs real money (~$2-4/hr smallest GPU droplet) and violates the no-paid-funding
  constraint. V1 relies on catalog + status-page RSS monitoring; genuine launchability
  detection deferred to V2 (possibly manual spot-checks).

**Oracle Cloud Infrastructure (OCI)**

- Signal A (free, authed): `ListShapes` per region/AD → catalog of BM.GPU.H100.8,
  BM.GPU.A100-v2.8, BM.GPU.B200.8 shapes (as they roll out). Returns per-shape
  `quota_names` and `is_flexible`.
- Signal B (free, authed): `GetComputeCapacityReport` → Oracle's official "is
  capacity available in this AD for this shape right now" signal. Cleanest
  launchability signal among hyperscalers — more direct than AWS Spot Placement
  Score. This is why adding OCI is worth the probe cost.
- **Known limitation:** Free tier does not include BM.GPU shapes; capacity report
  requires a paid tenancy. Mitigation: use a pay-as-you-go tenancy with a $0
  budget cap; capacity report API calls themselves are free. Document tenancy
  cost (~$0/month at rest) on Methodology page.

**Hut 8 Corp — catalog-only with manual attestation**

- Signal A (free, unauthed): scheduled fetch of `hut8.com/hpc-ai` (or current
  product page URL) every 24h. Parse visible SKU/pricing table.
- Signal B (quarterly, human-in-loop): email `ir@hut8.com` or public contact
  once per quarter asking for current Canadian regional GPU inventory. Response
  (or non-response) logged as an evidence record with `probe_type =
  manual_attestation`.
- **Known limitation:** No launchability signal. Verdict is always `listed` or
  `unknown`; never `phantom` unless Hut 8 themselves publish a "temporarily
  unavailable" indicator. Disclosed on every Hut 8 row.

**Iris Energy (IREN) — catalog-only with manual attestation**

- Signal A (free, unauthed): scheduled fetch of `iren.com/ai-cloud` (or current
  URL) every 24h. Parse visible shape/pricing information.
- Signal B (quarterly, human-in-loop): email to IR/sales same cadence as Hut 8.
- Same verdict limitations as Hut 8; disclosed per-row.

**Summary:** V1 produces genuine "listed vs launchable" verdicts for AWS, Azure,
GCP, OVH, and OCI. DigitalOcean, Hut 8, and Iris Energy get catalog-only or
catalog-plus-attestation verdicts in V1, with the limitation disclosed on every
affected row in the Phantom Wall. OCI's `GetComputeCapacityReport` is arguably
the cleanest launchability signal of any V1 provider and should be treated as
the gold-standard reference point on the Methodology page.

## Premises

1. **The phantom inventory gap is durable.** Providers will not close the
   listed-vs-launchable gap in the near term. The gap is the product.
2. **The policy audience engages with both live dashboards and formal reports.** Live
   dashboards provide screenshots and permalinks; quarterly reports provide citeable
   PDF/markdown artifacts. Both flow from the same evidence layer.
3. **G7 is the right peer group.** Comparing Canada to US, UK, France, Germany, Italy,
   Japan lands harder than OECD or "Canada vs. US" alone.
4. **Vercel + free API tiers are financially viable.** Probing cadence (every 30 min
   across ~15 providers × 7 countries) fits inside free tiers. No surprise bills.
5. **Live launchability probing is core, not optional.** Real probes — dry-run instance
   creates, Spot Placement Score calls, OVH stock-badge scraping, Azure SKU restrictions
   — are the moat. Without them this is another catalog comparison.
6. **Methodology transparency is a first-class feature.** The Methodology page is
   auto-generated from probe module docstrings. Anyone can audit exactly how a verdict
   was derived. Without this, MILA / CIFAR / ISED-adjacent researchers will not cite.
7. **Independent of government.** No ISED grants, no Alliance Canada coordination,
   no quiet embargoes. Journalism through software.
8. **The metric counts commercially rentable GPUs, not all GPUs on Canadian soil.**
   Alliance Canada (ex-Compute Canada), Mila's institutional compute, and ISED's
   AI Compute Access Fund allocations are real and meaningful — but they are not
   rentable by a Canadian startup or researcher without institutional access or
   a grant. The Sovereignty Gap Index question is "what can a builder get a
   credit card onto today," not "what hardware sits in Canada." The exclusion is
   disclosed explicitly on the Methodology page so policy readers don't mistake
   the boundary for an oversight.

## Cross-Model Perspective

A Claude subagent was dispatched with a clean briefing. Its independent read:

- **Key insight:** "Methodology transparency is a feature, not an appendix" IS the
  whole thing. Build the **evidence object** (timestamp, endpoint, request, response,
  verdict) as the core primitive. The dashboard, leaderboard, index, and reports are
  all views on top of that primitive. This reframes architecture from "dashboard with
  a hidden database" to "append-only evidence log with views on top."
- **Steelman:** The dashboard is a time-series claims registry. Each hour's probe
  results are signed, hash-chained, and committed to a public append-only log. A
  journalist, policy staffer, or lawyer can cite any data point with a commit-SHA
  permalink.
- **Existing reuse:** `github.com/vantage-sh/ec2instances.info` (MIT) covers the
  catalog layer for AWS/Azure/GCP. The novel 50% is the launchability probe layer
  (dry-runs, Spot Placement Score, quota introspection, stock-badge scrapers, verdict
  engine).
- **Challenged premise:** #2 (policy audience engages with live dashboards) is the
  weakest. ISED and CIFAR-adjacent folks consume PDFs and briefing notes, not live
  dashboards. Real pipeline: journalist → screenshot → minister's deck. Build the
  report pipeline as a first-class artifact.

**Response:** Premise 2 revised to include auto-generated reports alongside the live
dashboard. Both flow from the same evidence layer. The evidence-object primitive was
adopted as the architectural foundation (Approach B's structure).

## Approaches Considered

### Approach A: Phantom Wall First — Canada-only V1

Single Next.js page on Vercel. GitHub Actions cron probes AWS + Azure + OVH for Canada
only, every 30 minutes, commits evidence JSON to `/evidence/` in-repo. One view
(Phantom Wall). G7 Leaderboard, Timeline, Gap Index, reports layer in over weeks.

Effort: S (3-5 days). Risk: Low. Completeness: 4/10.

### Approach B: Evidence-First Monorepo — full scope, G7 day one (CHOSEN)

Monorepo with clean separation: `probes/` (per-provider workers), `evidence/` (append-only
log), `verdict/` (reconciliation engine), `web/` (Next.js dashboard), `reports/` (markdown
→ PDF generator). GitHub Actions matrix runs probes in parallel per provider. SQLite
(or LibSQL/Turso) for query layer, rebuildable from evidence log. All four views plus
auto-generated quarterly reports. G7 scope baked into probe configs from day one.

Effort: L (6-8 weeks for V1). Risk: Medium. Completeness: 9/10.

### Approach C: Registry as a Service

The primary product is a public, signed, hash-chained append-only evidence registry
exposed via REST API. Third-party sites, journalists, policy staff query the registry
directly. The dashboard is a thin consumer. Most defensible positioning; higher launch
bar.

Effort: L (2-3 weeks for MVP). Risk: Medium-High. Completeness: 8/10.

## Recommended Approach

**Approach B — Evidence-First Monorepo.** It honors the user's "everything at once"
scope preference while absorbing the subagent's evidence-object-as-primitive insight.
Approach A contradicts the stated scope. Approach C defers the visible artifact that
makes the policy pitch land.

### Core data type (defined once, used everywhere)

```typescript
interface EvidenceObject {
  timestamp: string;              // ISO 8601 UTC
  provider: 'aws' | 'azure' | 'gcp' | 'ovh' | 'digitalocean' | 'oci' | 'hut8' | 'iren';
                                  // V1 union. V2 will add 'ibm' | 'vultr' | 'paperspace' |
                                  // 'vast' | 'runpod' | 'lambda' | 'tensordock' | 'coreweave' —
                                  // scope-gated per §Provider Scope.
  country: 'CA' | 'US' | 'UK' | 'FR' | 'DE' | 'IT' | 'JP';
  region: string;                 // provider-native region identifier
  sku: string;                    // normalized canonical SKU (e.g., "H100-SXM-80GB")
  sku_raw: string;                // provider-native SKU string
  listed: boolean;
  launchable: boolean;
  verdict: 'phantom' | 'launchable' | 'inference_only' | 'quota_blocked' | 'unknown';
  // Verdict definitions:
  //   launchable     — SKU exists in region AND evidence indicates it can be launched
  //   phantom        — SKU listed in catalog but evidence indicates NOT launchable
  //                    (capacity exhausted, region not enabled, stock badge OUT)
  //   inference_only — SKU exists but provider documentation explicitly limits it to
  //                    serving/inference workloads (e.g., GCP A3 Edge). Detected via
  //                    per-provider allowlist in verdict rules.
  //   quota_blocked  — SKU offered by provider but the probing account has quota 0.
  //                    Informative but NOT phantom — inventory may exist; it's our
  //                    account that can't see it. Disclosed on Methodology page.
  //   unknown        — Probe failed (see §Error handling). Evidence record still written.
  probe_type: 'catalog_read' | 'dry_run_create' | 'spot_placement_score' | 'badge_scrape' | 'sku_restrictions' | 'capacity_report' | 'manual_attestation';
  endpoint: string;               // URL of the API or page probed
  request_hash: string;           // SHA-256 of the request payload
  response_hash: string;          // SHA-256 of the response payload
  response_excerpt: string;       // first 500 chars of response for debugging
  probe_duration_ms: number;
  probe_version: string;          // semver of the probe module that produced this
}
```

### Probe cadence, rate-limit budget, and data retention

- Catalog probes (unauthed pricing APIs): every 6 hours.
- Launchability probes (AWS Spot Placement Score, Azure SKU restrictions, OCI
  `GetComputeCapacityReport`): every 2 hours.
- OVH Playwright stock-badge scrape: every 4 hours (brittleness risk budgeted).
- GCP `instances.insert --validate-only` dry-runs: every 2 hours, 1 qps per zone.
- Hut8 + IREN public-page fetch: every 24 hours (page changes are rare; more
  frequent scraping looks like harassment and wastes budget).
- Hut8 + IREN manual attestation: once per quarter, human-in-loop email.
- **GitHub Actions budget audit** (2,000 min/month free tier):
    - Hyperscaler + OCI launchability jobs: 6 providers × 12 runs/day × 30 days
      × ~1 min each ≈ 2,160 min/month. Over budget if all run on GitHub-hosted.
    - Mitigation: consolidate into a single matrix workflow that fans out with
      `jobs.<id>.strategy.matrix`, amortizes container-start cost, and completes
      all providers in one billable minute per invocation. Target: ~720 min/month.
    - Playwright (OVH): 4h cadence × 30 days × ~2 min ≈ 360 min/month.
    - Page-fetch (Hut8, IREN): 24h × 30 days × 2 providers × ~30 sec ≈ 30 min/month.
    - **Projected total:** ~1,100 min/month, ~55% of free tier. Comfortable headroom.
    - If breached: move Playwright to a self-hosted runner on a home machine.
      Do NOT upgrade to paid Actions (violates §Constraints).

**Data retention — evidence log size management:**

Raw JSON commits at probe cadence would be ~30,000+ files/year. To avoid repo bloat:

- **Rolling 7-day raw window:** only the last 7 days of evidence stays as individual
  JSON files in `evidence/raw/`. The permalink page generates evidence permalinks
  against this window for fresh audit use.
- **Monthly compaction:** a nightly GitHub Actions job compacts >7-day-old evidence
  into a single month-partitioned Parquet file at `evidence/archive/YYYY-MM.parquet`.
  Individual commits preserved in Git history for cryptographic auditability;
  working-tree size stays bounded.
- **Git LFS:** Parquet archive files stored via Git LFS to keep the main working
  tree lean. **Sizing math** (revised for 8 providers):
    - Active (provider × country) pairs from the coverage matrix: **40**
      (Canada 8, US 8, UK 6, France 5, Germany 6, Italy 3, Japan 4).
    - Per pair, an average of **~3 current-gen SKUs** actually appear (not all 8
      in the taxonomy appear in every region).
    - Blended cadence across providers: ~8 probes/day/pair (launchability at 2h,
      catalog at 6h, Hut8/IREN pages at 24h averaged in).
    - Record count: 40 × 3 × 8 × 365 ≈ **~350k records/year**.
    - Per-record JSON ~1.5 KB including hashes + 500-char excerpt → **~525 MB/year
      raw**. Parquet with ZSTD columnar compression lands at ~10–15% of raw →
      **~55–80 MB/year archived**.
    - GitHub LFS free tier (1 GB storage, 1 GB/mo bandwidth) fits **~12 years**
      before pruning is necessary. If the archive outgrows the free tier earlier
      (e.g., we expand to V2 providers), the pruning policy is: drop raw
      evidence older than 5 years, keep quarterly summary rollups only.
- **Permalink format:** `/evidence/{sha}/{path}` resolves against Git history, so
  old evidence remains permalinked even after archival.

### Error handling and failure semantics

Every probe run produces exactly one outcome per (provider, region, SKU) triplet,
even on failure. This keeps the evidence log dense and analysis-friendly.

- **Probe success (API 2xx):** `verdict` is one of `launchable`, `phantom`,
  `inference_only`, `quota_blocked`.
- **Probe failure (API 5xx, network, auth expiry):** `verdict = unknown`, with
  `response_excerpt` containing the error message and `probe_duration_ms` reflecting
  time-to-failure. Three consecutive `unknown` verdicts for the same triplet raise
  a "degraded probe" alert that surfaces on the Methodology page.
- **Scraper breakage (OVH badge selector drift):** `verdict = unknown`, subtype
  `scraper_drift`. Methodology page lists affected providers until fix.
- **Retry policy:** one in-run retry with 15s backoff. No exponential backoff; the
  30m/2h/4h cadence naturally retries the next window.
- **No silent skips.** A failed probe always produces an evidence record.

### Auth strategy

- **Unauthed probes** (OVH catalog, OVH badge scrape, Hut 8 page fetch, IREN
  page fetch, Azure Retail Prices): run on any runner, no secrets.
- **Authed probes** (AWS, Azure SKU restrictions, GCP, DO, OCI): one read-only
  service account per provider. Credentials stored in GitHub Secrets. Each
  account uses a dedicated email owned by the project, not a personal account.
- **No paid API keys.** If a probe requires a paid API call, skip that provider.
  OCI's tenancy is pay-as-you-go but the `GetComputeCapacityReport` API call
  itself is free and the tenancy at-rest cost is ~$0 with a $0 budget cap —
  this is the single acceptable exception.

### Rebuild-from-evidence property

Because the evidence log is append-only and the verdict engine is deterministic,
the SQLite cache can be rebuilt from evidence at any time. This is the reproducibility
guarantee: anyone can clone the repo, run `pnpm rebuild`, and produce the same state.

### Rebuild-from-evidence determinism

The SQLite cache can be rebuilt from evidence, but only for a pinned
`taxonomy_version` and `verdict_engine_version`. Replaying historical evidence with
a newer engine produces *current-interpretation* verdicts, not *historical* verdicts.
The evidence log itself is immutable; interpretation is versioned. Both the
dashboard header and every report include `taxonomy_version` + `engine_version`
badges so readers know what interpretation they're seeing.

## What I noticed about how you think

- **You consistently asked for the complete thing, not the wedge.** "A combination of all of
  these options," "everything at once," "all four views." That's ambition-scale thinking.
  The evidence-object architecture is the unlock that makes the full scope buildable
  from one pipeline — without it, ambition + scope would mean four disjoint features
  and a painful launch.

- **You dropped the Pu reference for audience reasons, not inspiration reasons.** "It's
  not widely enough known to be relevant." You were already running the audience's
  mental model, not the builder's. That instinct matters — it's the difference between
  a project that speaks to its target and one that talks about its origin story.

- **You asked me to clarify twice instead of picking a weak option.** Most people
  pick the nearest option to move forward. You held the room until the framing was
  right. That's what good product discipline looks like on a team.

- **You picked "nothing exists" as the positioning claim without hedging.** Bold, and
  partially defensible: pricing comparison is crowded, sovereign AI discourse is
  crowded, but the intersection plus phantom-inventory exposure really is empty. The
  stronger framing: "nothing quite like this exists" — which leaves room to
  acknowledge and learn from prior-adjacent tools (Vantage, Our World in Data, Stanford
  AI Index) rather than waving them off.
